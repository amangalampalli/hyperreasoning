# Dataset Format

This project uses synthetic offline RL transition data generated by `scripts/data/collect_data.py`.

## Location

Runs are stored under:

```text
artifacts/synthetic/<run_id>/
```

Each run contains:

- `dataset.jsonl`
- `run_summary.json`
- per-task directories containing:
  - `plan_bank.json`
  - `summary.json`
  - `transitions.jsonl`
  - `episodes/<episode_id>.json`

## Main Training Input

The canonical offline bootstrap input is:

```text
artifacts/synthetic/<run_id>/dataset.jsonl
```

The trainer loads one or more run directories and reads their `dataset.jsonl` files.

## Transition Shape

Raw collected transitions contain:

- `state`
- `action`
- `reward`
- `next_state`
- `done`
- `valid_actions`
- `info`

During dataset normalization in `data/datasets.py`, these become canonical transitions with:

- `obs`
- `action`
- `action_id`
- `reward`
- `next_obs`
- `terminated`
- `truncated`
- `action_mask`
- `next_action_mask`
- `task_id`
- `metadata`
- `discount`
- `n_steps`

## State Contents

The current collected state includes compact branch-control information such as:

- `task_id`
- `family`
- `current_bank_id`
- `current_depth`
- `current_strategy`
- `current_heuristic_score`
- `current_compile_success`
- `current_visible_test_passed`
- `path_length`
- `remaining_steps`
- `visible_child_count`
- `hidden_child_count`
- `known_result_count`
- `best_compile_success`
- `best_visible_test_passed`
- `compile_budget_remaining`
- `child_slots`
- `valid_actions`

## Action Space

The canonical action table is defined in `env/dsl_env.py`.

Current actions:

- `SELECT_CHILD_0`
- `SELECT_CHILD_1`
- `SELECT_CHILD_2`
- `SELECT_CHILD_3`
- `REQUEST_MORE_CANDIDATES`
- `REFINE_CURRENT_PLAN`
- `COMPILE_TO_CODE`
- `BACKTRACK`
- `TERMINATE`
- `REPAIR_FROM_FEEDBACK`

`REPAIR_FROM_FEEDBACK` exists in the canonical training/eval action table even if older collected runs do not contain that action yet.

## Label Types

`info.label_tier` identifies how strong the grounding is for a transition:

- `dsl_only`
  - purely search/control data
- `compile`
  - compile-verified branch outcome
- `visible_test`
  - compile + visible-test grounded branch outcome

This means the dataset is usually a mixture of:

- many cheap search-control transitions
- fewer compile-labeled transitions
- even fewer visible-test-labeled transitions

## N-step Support

The trainer supports:

- direct 1-step loading from `dataset.jsonl`
- optional n-step reconstruction from per-episode JSON files through `data/replay_dataset.py`

## Practical Guidance

- Use bulk heuristic runs for broad search-control coverage.
- Use hybrid/verifier-heavy runs to add stronger reward grounding.
- Mix both during offline Rainbow training rather than relying on only one run type.
- Use deterministic task manifests in `data/splits/` for train/eval separation.
